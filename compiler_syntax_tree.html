Core concepts

Mathematical foundations

A compiler's front-end transforms raw source text into a structured Abstract Syntax Tree (AST) using a formal grammar G = (VN, VT, P, S) and recursive descent parsing.

📐

Formal grammar G(V,T,P,S)

A context-free grammar defines every valid program. Non-terminals derive into sequences of terminals and other non-terminals following production rules.

G = (VN, VT, P, S) — CFG definition
🔁

Recursive descent parser

Each non-terminal maps to a parsing function that calls itself or peers recursively. Mirrors the grammar structure directly — elegant and maintainable.

parseExpr() → parseTerm() → parseFactor()
🌲

Abstract syntax tree

The AST captures program structure without redundant grammar symbols. Each internal node is an operator; leaf nodes are literals and identifiers.

depth(AST) = O(log n) — balanced expressions
✓

Syntax correctness

The parser enforces grammar production rules. Mismatched tokens, missing operands, or unclosed parentheses trigger precise, actionable error messages.

∀ token t ∈ First(A) → predict production
⚡

Parse table optimisation

LL(1) parse tables allow O(n) deterministic parsing. The parser looks ahead exactly one token to choose the unique matching production rule.

T[A,a] = p iff a ∈ First(α) — LL(1) table
📊

Memoisation (packrat)

Cache parse results at each (position, rule) pair. Eliminates exponential blowup in ambiguous or backtracking grammars, guaranteeing linear time.

memo[pos][rule] → O(n·|G|) total work

Algorithm details

Each algorithm addresses a distinct stage in transforming source text into a verified abstract syntax tree.

Recursive descent parsing

Application: For each non-terminal in the grammar, a dedicated function implements the production rule. The call stack mirrors the derivation tree — making the algorithm intuitive, debuggable, and easy to extend.

Recursive descent works top-down: starting from the start symbol, it predicts which production to use by examining the next token (lookahead). Each call either succeeds and consumes tokens, or throws a parse error.

Time: O(n) — LL(1) grammar Space: O(d) — recursion depth
  • Every non-terminal in the grammar has one parse function
  • Lookahead token selects the correct production rule
  • Left recursion must be eliminated before parsing
  • Call stack depth equals parse tree depth
  • Error messages pinpoint the exact unexpected token
  • Easy to add semantic actions (type-checking, codegen)
JavaScriptrecursive_descent.js
// Recursive descent parser for arithmetic grammar
// Grammar: Expr → Term (('+' | '-') Term)*
//          Term  → Factor (('*' | '/') Factor)*
//          Factor → '(' Expr ')' | Unary
//          Unary  → '-' Unary | Primary
//          Primary → NUMBER | IDENT | Assign

function parseExpr(depth) {
  calls++;
  let left = parseTerm(depth + 1);
  while (cur().val === '+' || cur().val === '-') {
    const op = consume();
    const right = parseTerm(depth + 1);
    left = node('BinOp', op.val, [left, right]);
  }
  return left;
}

function parseTerm(depth) {
  calls++;
  let left = parseFactor(depth + 1);
  while (cur().val === '*' || cur().val === '/') {
    const op = consume();
    left = node('BinOp', op.val, [left, parseFactor(depth+1)]);
  }
  return left;
}

function parseFactor(depth) {
  calls++;
  if (cur().val === '(') {
    consume('(');
    const e = parseExpr(depth + 1);
    consume(')'); // enforce closing paren
    return e;
  }
  return parseUnary(depth + 1);
}

Grammar analysis — First & Follow

Application: Computing FIRST and FOLLOW sets for each non-terminal is the prerequisite for constructing an LL(1) parse table. FIRST tells us which tokens can begin a derivation; FOLLOW tells us which tokens can appear after a non-terminal.

If FIRST and FOLLOW sets are disjoint for all alternatives of a non-terminal, the grammar is LL(1) — the parser can always make a deterministic choice using just one token of lookahead.

FIRST: O(|P| · |Σ|) Table: O(|N| × |Σ|)
  • FIRST(α) — terminals that begin strings derived from α
  • FOLLOW(A) — terminals that can follow A in a sentential form
  • LL(1) condition: no table entry has more than one production
  • Left recursion elimination required before FIRST computation
  • Left factoring resolves common prefixes in alternatives
  • FOLLOW(S) always contains $ (end of input)
JavaScriptfirst_follow.js
// Compute FIRST sets for all non-terminals
function computeFirst(grammar) {
  const first = {};
  for (const nt of grammar.nonTerminals) {
    first[nt] = new Set();
  }
  let changed = true;
  while (changed) {
    changed = false;
    for (const { lhs, rhs } of grammar.productions) {
      const before = first[lhs].size;
      // Add FIRST of each symbol until no ε
      for (const sym of rhs) {
        if (isTerminal(sym)) {
          first[lhs].add(sym); break;
        }
        setUnion(first[lhs], first[sym]);
        if (!first[sym].has('ε')) break;
      }
      if (first[lhs].size !== before) changed = true;
    }
  }
  return first;
}

// Build LL(1) parse table T[A, a]
function buildParseTable(grammar, first, follow) {
  const table = {};
  for (const { lhs, rhs } of grammar.productions) {
    const fst = firstOfString(rhs, first);
    for (const a of fst) {
      if (a !== 'ε') table[lhs][a] = rhs;
    }
    if (fst.has('ε'))
      for (const b of follow[lhs]) table[lhs][b] = rhs;
  }
  return table; // conflict → not LL(1)
}

Error detection & recovery

Application: A production-quality parser should not halt at the first error. Panic-mode recovery synchronises on FOLLOW sets to skip tokens and attempt to resume parsing, reporting multiple errors in a single pass.

Error messages should identify the unexpected token, the expected set, and the source location. An error node is inserted into the AST so later compiler phases can continue analysis.

Recovery: O(n) skip Error nodes in AST
  • Unexpected token → report error with position and expected set
  • Panic mode: skip tokens until synchronisation set (FOLLOW) is found
  • Insert ErrorNode into AST to allow semantic analysis to continue
  • Phrase-level recovery corrects small mistakes (missing semicolons)
  • Error productions handle common idioms like missing operands
  • Limit cascading errors with a max-error threshold
JavaScripterror_recovery.js
// Error recovery with panic-mode synchronisation
class ParseError extends Error {
  constructor(msg, token, expected) {
    super(msg);
    this.token    = token;
    this.expected = expected;
    this.pos      = token.pos;
  }
}

function synchronise(parser, syncSet) {
  // Consume tokens until sync point
  while (!parser.isAtEnd()) {
    if (syncSet.has(parser.cur().type)) return;
    parser.advance();
  }
}

function parsePrimaryWithRecovery(parser) {
  try {
    return parser.parsePrimary();
  } catch (e) {
    if (e instanceof ParseError) {
      parser.errors.push(e);
      synchronise(parser, EXPR_FOLLOW_SET);
      // Insert error node — AST remains valid
      return { type: 'Error', msg: e.message,
               pos: e.pos, children: [] };
    }
    throw e;
  }
}

Parse speed optimisation

Application: Packrat parsing memoises every (rule, position) pair. Each combination is computed at most once, converting potentially exponential backtracking into guaranteed O(n·|G|) linear time with a small constant factor.

For large files, lexer-level optimisations (DFA-based scanner, zero-copy string views) and AST arena allocation can reduce parse time by a further 2–5×.

Packrat: O(n · |G|) Naive: O(2ⁿ) worst case
  • Packrat: cache[pos][rule] eliminates duplicate sub-parses
  • LL(1) table look-up is O(1) per token — avoids backtracking entirely
  • Lazy tokenisation: lex on demand, not upfront
  • Arena allocation: bump-allocator for AST nodes (cache-friendly)
  • Zero-copy string views: source slices instead of allocations
  • Incremental re-parse: only re-parse changed subtrees (LSP)
JavaScriptpackrat_memo.js
// Packrat memoisation — O(n·|G|) guaranteed
class PackratParser {
  constructor(src) {
    this.src  = src;
    this.memo = new Map(); // key: `${rule}:${pos}`
    this.hits = 0;
    this.misses = 0;
  }

  apply(rule, pos) {
    const key = `${rule}:${pos}`;
    if (this.memo.has(key)) {
      this.hits++;
      return this.memo.get(key); // cache hit
    }
    this.misses++;
    const result = this.rules[rule](this, pos);
    this.memo.set(key, result);
    return result;
  }

  hitRate() {
    const total = this.hits + this.misses;
    return total ? (this.hits / total * 100).toFixed(1) + '%' : '0%';
  }
}

Formal grammar model

A context-free grammar G = (VN, VT, P, S) generates the language of all valid programs. Every parse tree represents a complete derivation from the start symbol S to a string of terminals.

VNNon-terminals — parser function names (Expr, Term, Factor…)
VTTerminals — actual tokens from the lexer (NUM, IDENT, +, -, *)
PProduction rules — define valid substitutions for each non-terminal
SStart symbol — the root non-terminal (typically Program or Expr)
FIRST(A)Terminals that can start a string derived from non-terminal A
FOLLOW(A)Terminals that can immediately follow A in any sentential form
NullableNon-terminal A from which ε (empty string) can be derived
LL(1)Grammar parseable left-to-right with 1 token lookahead, no backtrack
AST nodeInternal = operator, leaf = operand; grammar noise discarded
Parse errorToken not in FIRST set of any applicable production
